\section{Data Logic Layer}

After presenting the main functionalities of the application, this section
focuses on the data logic layer, which is responsible for modeling and
managing the information handled by the system.
In particular, the structure of the database and the relationships between
entities are described through an Entity-Relationship schema.

\subsection{Entity-Relationship Schema}

\begin{figure}[H]
    \centering
    \fbox{\includegraphics[width=0.95\textwidth]{diagrams/ERScheme}}
    \caption{Entity-Relationship Schema of the Lecture Notes App}
    \label{fig:er-schema}
\end{figure}

The database is designed to represent the core concepts of the application through a set of entities and relationships. The main entities are: \textit{User}, \textit{Note}, and \textit{Course}. The system also includes relationships between these entities, namely \textit{Uploads}, \textit{Favorite}, \textit{Rating}, and \textit{Belongs to}.
In particular, \textit{Favorite} and \textit{Rating} are associative relationships that model many-to-many interactions between users and notes.
The relationships between these elements can be summarized as follows:
\begin{itemize}
    \item A User can upload multiple Notes (1:N relationship)
    \item A User can mark multiple Notes as Favorite (M:N relationship)
    \item A User can assign at most one Rating to each Note, while being able to rate multiple Notes (M:N relationship with constraint of uniqueness)
    \item Each Note belongs to exactly one Course (N:1 relationship)
\end{itemize}
The ER diagram in Figure~\ref{fig:er-schema} provides a visual representation of the database structure and highlights these relationships.

\subsection{Entity and Relationship Description}

This section describes both the entities and relationships of the system, along with their mapping to the relational database schema.

\paragraph{User}
The \textit{User} entity stores authentication credentials and profile information, including: \texttt{id}, \texttt{username}, \texttt{email}, \texttt{password}, and \texttt{createdAt}.

\paragraph{Course}
The \textit{Course} entity represents a subject or course and includes: \texttt{id} and \texttt{name}.

\paragraph{Note}
The \textit{Note} entity contains the main information related to uploaded lecture notes, including: \texttt{id}, \texttt{author\_id}, \texttt{course\_id}, \texttt{title}, \texttt{description}, \texttt{upload\_date}, and \texttt{file\_path}.
From the relational perspective:
\begin{itemize}
    \item The \texttt{author\_id} attribute is a foreign key referencing \texttt{User(id)}, implementing the \textit{Uploads} relationship.
    \item The \texttt{course\_id} attribute is a foreign key referencing \texttt{Course(id)}, implementing the \textit{Belongs to} relationship.
\end{itemize}

These relationships are therefore represented through foreign keys and do not require separate tables.

\paragraph{Uploads (Relationship)}
The \textit{Uploads} relationship models the association between a User and the Notes they upload. It is a one-to-many (1:N) relationship and is implemented in the relational schema through the foreign key \texttt{author\_id} in the \textit{Note} table. As a result, no separate table is required.

\paragraph{Favorite (Relationship)}
The \textit{Favorite} relationship models the association between users and their favorite notes. Since it is a many-to-many relationship, it is mapped to a separate relational table including: \texttt{user\_id}, \texttt{note\_id} and \texttt{created\_at}.

\paragraph{Rating (Relationship)}
The \textit{Rating} relationship represents the evaluation given by a user to a note. This many-to-many relationship is also mapped to a separate table including: \texttt{id}, \texttt{user\_id}, \texttt{note\_id}, \texttt{value}, and \texttt{created\_at}.
A uniqueness constraint on (\texttt{user\_id}, \texttt{note\_id}) ensures that each user can assign only one rating per note.

\paragraph{Belongs to (Relationship)}
The \textit{Belongs to} relationship associates each \textit{Note} with a specific \textit{Course}. It is a many-to-one (N:1) relationship and is implemented through the foreign key \texttt{course\_id} in the \textit{Note} table. This relationship is therefore not mapped to a separate table.
